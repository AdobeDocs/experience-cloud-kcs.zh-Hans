# 优化 [!DNL Dispatcher] 缓存

## 描述


本文重点介绍AEM Dispatcher中的最新优化以及如何最好地利用这些优化。  AEM Dispatcher是缓存 [反向代理](https://stackoverflow.com/questions/224664/difference-between-proxy-server-and-reverse-proxy-server) 专为与Adobe Experience Manager一起使用而设计的服务器。  它可以作为模块安装并在现有的Web服务器软件中运行。  在撰写本文时， [支持dispatcher模块](https://helpx.adobe.com/experience-manager/dispatcher/using/dispatcher-install.html) on [!DNL Apache HTTP Server]、 Microsoft IIS和 [!DNL iPlanet].


## 分辨率

<br><br>调度程序缓存的工作原理<br><br><br><br><br><br>
在最基本的级别上，AEM Dispatcher是反向代理，它通过执行缓存、缓存刷新和缓存失效来工作。

有关调度程序的更多详细信息，请参阅相关链接：

1. [调度程序的工作方式及其安装方法](https://helpx.adobe.com/experience-manager/dispatcher/using/dispatcher.html).
2. [调度程序中可用的配置选项](https://helpx.adobe.com/experience-manager/dispatcher/using/dispatcher-configuration.html).
3. [有关调度程序工作原理的网络研讨会](https://github.com/cqsupport/webinar-dispatchercache)  — 请注意，演示文稿中的某些信息基于Dispatcher的旧版本。
4. [有关调度程序功能、CDN使用和安全性的Gems网络研讨会](https://docs.adobe.com/ddc/en/gems/dispatcher-caching---new-features-and-optimizations.html).
5. [Dispatcher中新增功能的Gems会议（v4.1.9之后）](https://helpx.adobe.com/experience-manager/kt/eseminars/gems/aem-dispatcher.html).

<br><br><br><br><br><br>优化调度程序缓存<br><br><br><br><br><br>
以下是优化调度程序缓存的一些方法：

1. <b>缓存几乎所有内容</b>  — 这意味着缓存用户请求多次的任何内容。
2. <b>缓存不同时间段的个性化内容</b>  — 如果您的网站具有个性化内容，请考虑使用 [[!DNL Apache Sling Dynamic Includes]](https://helpx.adobe.com/experience-manager/kt/platform-repository/using/sling-dynamic-include-technical-video-setup.html) 在AEM应用程序中利用 [!DNL Ajax] （浏览器级别的异步JavaScript和XML调用）、 SSI（Web服务器级别的服务器端包含）和ESI（CDN级别的边缘端包含），以便在不同时间段内缓存页面的不同部分。
3. <b>切勿删除实时调度程序上的调度程序缓存</b>  — 如果调度程序提供实时内容并且您删除了缓存，则会导致大量请求返回AEM。  因此，绝不应删除实时调度程序上的调度程序缓存。
4. <b>设置缓存的最优值 </b> — 在删除 [!DNL Dispatcher] 缓存，提取 [!DNL Dispatcher] 在负载平衡器上，删除缓存，然后运行web crawler工具以在调度程序上缓存文件，然后再将其置于负载平衡器上。
5. <b>缓存错误页</b>  — 利用 <b>[DispatcherPassError 1](https://helpx.adobe.com/experience-manager/dispatcher/using/dispatcher-install.html#ApacheWebServer) </b>([!DNL Apache] 特定于Web服务器的)指令，用于从调度程序缓存中提供错误页面（如404）。
6. <b>GZip压缩所有文件类型，但预压缩的文件类型除外 </b> — 在 [!DNL Apache] Web服务器、 [mod_deflate](https://httpd.apache.org/docs/2.4/mod/mod_deflate.html) 可以使用，但请确保 <b>`Vary: User-Agent` </b>标题<b> </b>未设置。  在Microsoft IIS中，使用 [动态压缩](https://docs.microsoft.com/en-us/iis/configuration/system.webserver/httpcompression/).

   [!DNL Apache] 配置示例（仅指定某些内容类型以避免预压缩文件类型）：

   `AddOutputFilterByType DEFLATE text/html text/plain text/xml text/css text/javascript application/javascript`
7. <b>启用 [/serveStaleOnError](https://helpx.adobe.com/experience-manager/kb/ServeStaleContentOnError.html)</b> 在 `/cache` 配置 — 当AEM实例出现错误时，提供旧缓存文件。
8. <b>添加 [/gracePeriod](https://docs.adobe.com/content/help/en/experience-manager-dispatcher/using/configuring/dispatcher-configuration.html#configuring-the-dispatcher-cache-cache)</b> 到 `/cache` 配置 — 定义在上次内容发布事件（“激活”）后，缓存中仍可能提供失效的失效资源的秒数。  这会减少在大型内容发布活动（如“树激活”）期间返回到发布实例的请求数。
9. <b>将规则添加到 [/ignoreUrlParams](https://helpx.adobe.com/experience-manager/dispatcher/using/dispatcher-configuration.html#IgnoringURLParameters)</b>  — 忽略应用程序不需要或使用的querystring参数。  这允许缓存URL，即使存在查询字符串也是如此。
10. <b>缓存Cache-Control和Last-Modified响应标头</b>  — 使用<b> [/headers](https://helpx.adobe.com/experience-manager/dispatcher/using/dispatcher-configuration.html#CachingHTTPResponseHeaders)</b> 缓存HTTP响应标头的配置 <b>`Cache-Control`</b> 和 <b>`Last-Modified` </b>(和/或 <b>`ETag`</b> 标头)。  这有助于简化和优化CDN和浏览器级别的缓存。  缓存这些标头会使其只设置AEM的标头，而不是Web服务器本身。  请注意，执行此操作时，将需要开始从AEM应用程序发送标头。
11. <b>尽可能长地缓存内容</b> 和 <b>减少返回AEM的请求</b>  — 通过启用 [重新获取](https://helpx.adobe.com/experience-manager/kb/optimizing-the-dispatcher-cache.html#refetching-flush) 刷新所有刷新代理。  或使用 [<b>/enableTTL</b>](https://helpx.adobe.com/experience-manager/kb/optimizing-the-dispatcher-cache.html#use-ttls) 设置 <b>`Cache-Control: max-age=...`</b> 标头来缓存文件。  请参阅 [下面](https://helpx.adobe.com/experience-manager/kb/optimizing-the-dispatcher-cache.html#use-ttls) 以了解有关此主题的详细信息。

<br><br><br><br><br><br>使用TTL<br><br><br><br><br><br>
截至 [!DNL Dispatcher] 版本4.1.11, [/enableTTL 1](https://helpx.adobe.com/experience-manager/dispatcher/using/dispatcher-configuration.html#ConfiguringTimeBasedCacheInvalidationenableTTL) 可以在 `.any` 文件配置。  此设置使调度程序遵循HTTP中设置的缓存过期时间 `Cache-Control` 响应标头。  换言之，调度程序的功能将与CDN类似，即文件过期时，会出现主要形式的缓存失效。  实施此功能并开始发送后 <b>`Cache-Control: max-age=...` </b>对于来自AEM的所有响应，您可以在publish实例中安全地禁用调度程序刷新代理。

在禁用发布实例上的刷新代理后，您可能仍希望能够刷新调度程序缓存。  在这种情况下，您可以使用 [ACS Commons — 调度程序刷新UI](https://adobe-consulting-services.github.io/acs-aem-commons/features/dispatcher-flush-ui/index.html).  此工具安装在创作实例上。  它为用户提供了一个用户界面，用户可以在其中执行手动缓存刷新请求。

<b>I.启用TTL（“生存时间”或到期）样式无效的步骤：</b>

1. 在AEM应用程序中修改源代码以发送 <b>`Cache-Control` </b>标题和 <b>`Last-Modified` </b>对于所有尚未设置的请求。
2. 安装 [!DNL Dispatcher] 4.1.11或更高版本。
3. 已设置 <b>[/enableTTL 1](https://helpx.adobe.com/experience-manager/dispatcher/using/dispatcher-configuration.html#ConfiguringTimeBasedCacheInvalidationenableTTL)</b> 在 `.any` 站点的场配置。
4. 设置 <b>[/headers](https://helpx.adobe.com/experience-manager/dispatcher/using/dispatcher-configuration.html#CachingHTTPResponseHeaders) </b>缓存的配置 <b>`Cache-Control`</b> 和 <b>`Last-Modified`</b> 标题。
5. 重新启动 Web 服务器。


<b>二。 在发布实例上禁用调度程序刷新代理：</b>

调度程序现在将使用 `Cache-Control` 用于控制缓存文件失效的标头。  因此，不再需要从发布实例中刷新调度程序。

1. 转到 `/etc/replication/agents.publish.html` 在每个发布实例上。
2. 转到每个刷新代理的配置并禁用代理。


<b>三。 允许来自创作实例的手动调度程序刷新请求：</b>

既然已禁用刷新代理，您将完全依赖 <b>`Cache-Control` </b>用于控制调度程序上内容何时刷新的标题。  您仍然可以允许用户手动刷新调度程序缓存：

1. 安装 [ACS Commons — 调度程序刷新UI](https://adobe-consulting-services.github.io/acs-aem-commons/features/dispatcher-flush-ui/index.html) 在创作实例上。
2. 在创作实例上配置刷新代理。
3. 在每个代理配置中，设置 <b>触发器</b> = <b>忽略默认值 </b>选项。 当用户单击 <b>（取消）发布 </b>或 <b>（取消）激活</b> 在AEM UI中。

<br><br><br><br><br><br>重新获取调度程序刷新<br><br><br><br><br><br>
要优化调度程序刷新请求，所有调度程序刷新代理都应启用称为重新获取刷新的功能。

要启用重新获取调度程序刷新，请执行以下操作：

1. 转到 <b>*http://aemhost:port*/crx/packmgr/index.jsp</b> 并以管理员身份登录。
2. 从下载包 [此处](https://github.com/cqsupport/webinar-dispatchercache/blob/master/packages/dispatcher-flush-refetch-samplecode-1.0.zip?raw=true).
3. 将包上载并安装到包管理器。
4. 转到调度程序刷新代理配置。 例如 <b>/etc/replication/agents.author/flush.html</b>
5. 单击 <b>[!UICONTROL 编辑]</b>
6. 设置以下内容
   - <b>序列化类型</b> = <b>重新获取调度程序刷新</b>
   - <b>扩展</b> = <b>HTTP方法</b> = <b>POST</b>
7. 单击 <b>[!UICONTROL 保存]</b>


请注意，上面安装的软件包只是一个基本示例。  要自定义和优化重新获取刷新，您可以修改其发送的URI列表。  该代码是开源代码，可在 [此处](https://github.com/cqsupport/webinar-dispatchercache/tree/master/src/refetching-flush-agent/refetch-bundle).  该代码将URI列表作为参数添加到请求正文中，以告知调度程序要重新获取的路径。  您可以根据应用程序要求添加更多路径以优化网站的缓存功能。
<br><br><br><br><br><br>重新获取刷新的更详细说明<br><br><br><br><br><br>
调度程序刷新通常通过删除文件来工作：

1. 触控 `.stat` 文件
2. 删除 `/content/foo.\*`
3. 删除 `/content/foo/_jcr_content`


由于文件在步骤2中被删除，下次用户请求类似 `/content/foo.html` 或 `/content/foo.json`，则在“重新获取”文件时，对同一文件的后续请求也将发送到publish实例，直到文件被缓存为止。  如果响应缓慢或流量过大（如主页），可能会导致发布实例层发生洪泛。

要解决此问题，请启用调度程序的一项称为重新获取的功能。  此功能允许您发送一个URI列表，调度程序应主动“重新获取”并替换该列表，而不是删除。
